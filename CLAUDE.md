## Project state

This is a [T3 Stack](https://create.t3.gg/) app (Next.js App Router + Prisma + Tailwind) generated by
`create-t3-app` (`ct3aMetadata.initVersion: 7.40.0`). It is currently just the scaffold: a single `Post`
model, the default T3 landing page (`src/app/page.tsx`), no tRPC router, no NextAuth setup, and no tests,
despite tRPC/NextAuth/Drizzle being listed as stack options in README.md — do not assume those are wired up;
check `package.json` before relying on them.

## Commands

- `npm run dev` — start dev server (Next.js with Turbopack)
- `npm run build` / `npm run start` — production build / run
- `npm run preview` — build then start (prod-mode smoke test)
- `npm run check` — lint + typecheck (`next lint && tsc --noEmit`); run this before considering a change done
- `npm run lint` / `npm run lint:fix` — ESLint only
- `npm run typecheck` — `tsc --noEmit` only
- `npm run format:check` / `npm run format:write` — Prettier (with `prettier-plugin-tailwindcss` for class sorting)
- There is no test runner configured in this repo (no test script, no test framework dependency).

### Database (Prisma / Postgres)

- `./start-database.sh` — spins up a local Postgres in Docker/Podman using `DATABASE_URL` from `.env`
  (bash script; on Windows run it from WSL). Requires `.env` to exist (copy from `.env.example` first).
- `npm run db:generate` — `prisma migrate dev` (create/apply a migration in dev and regenerate the client)
- `npm run db:push` — `prisma db push` (push schema without a migration, for quick iteration)
- `npm run db:migrate` — `prisma migrate deploy` (apply existing migrations, for prod/CI)
- `npm run db:studio` — open Prisma Studio
- `postinstall` runs `prisma generate` automatically after `npm install`.

## Architecture

- **Prisma client output location is non-default**: `prisma/schema.prisma` has
  `generator client { output = "../generated/prisma" }`, so the generated client lives at
  `generated/prisma/`, not `node_modules/@prisma/client`. Import it via the shared singleton at
  `src/server/db.ts` (`import { db } from "@/server/db"`), which handles the dev hot-reload
  global-singleton pattern — never instantiate `PrismaClient` directly elsewhere. Note `generated/prisma`
  is currently committed to git (not gitignored) — check whether that's intentional before assuming it
  should be regenerated-and-ignored.
- **Env validation**: all environment variables must be declared in the Zod schemas in `src/env.js`
  (`server` vs `client`, the latter requiring a `NEXT_PUBLIC_` prefix) via `@t3-oss/env-nextjs`. Adding a
  raw `process.env.FOO` reference elsewhere bypasses this validation — add new vars to `src/env.js` and to
  `.env.example` instead. `next.config.js` imports `src/env.js` at build time so an invalid/missing env var
  fails the build (set `SKIP_ENV_VALIDATION=1` to bypass, e.g. for Docker builds).
  Env access from a Server Component (e.g. `src/app/**`) can use `@/env` directly; client components
  can only see `NEXT_PUBLIC_`-prefixed vars.
- **Path alias**: `@/*` maps to `src/*` (see `tsconfig.json`); use it instead of relative `../../` imports.
- **Routing**: Next.js App Router under `src/app/`. `src/app/layout.tsx` loads the `Geist` font and global
  styles (`src/styles/globals.css`, Tailwind v4 via `@tailwindcss/postcss`).
- **Linting**: flat ESLint config (`eslint.config.js`) extends `next/core-web-vitals` plus
  `typescript-eslint`'s `recommendedTypeChecked`/`stylisticTypeChecked` (type-aware — requires the
  TS project service, so lint can be slower and needs valid `tsconfig.json` coverage). Notable rule
  deviations: `consistent-type-imports` is enforced (warn, inline style), unused args prefixed with `_`
  are allowed, `no-misused-promises` is relaxed for JSX attributes (so `onClick={async () => ...}` is fine).

## Product direction (from `.stitch/`)

`.stitch/` holds [Stitch](https://stitch.withgoogle.com/)-generated design mockups — **not wired into the
Next.js app** (no imports from `src/`, plain standalone HTML with the Tailwind CDN script and Google
Fonts, not the project's Tailwind v4 setup). Treat them as the visual/product spec to implement against,
not as source code to run as-is.

"Hajjem" is a mobile-first (dark-mode-by-default, PWA-oriented) barber-booking app for the Tunisian market
(French UI copy), with two user roles: **clients** who search for and book barbers, and **barbers** who
manage their business. Mockup screens, one folder per screen (`code.html` + `screen.png`):

- `hajjem_accueil` — client home/landing screen
- `connexion_hajjem` — login screen (shared client/barber)
- `inscription_client_hajjem` — client signup screen
- `inscription_barbier_hajjem` — barber signup screen
- `trouver_un_coiffeur` — client search/discovery ("Rechercher par ville ou nom...", nearby barber list)
- `profil_barber` — public barber profile (services list + prices, booking flow)
- `gestion_des_rendez_vous` — client's appointments list ("Mes Rendez-vous")
- `tableau_de_bord_barber` — barber's dashboard (today's / upcoming appointments), desktop layout
- `tableau_de_bord_barber_mobile` — barber's dashboard, mobile layout
- `mon_profil_professionnel` — barber's own profile editor
- `espace_barber_plans` — barber subscription/pricing plans (Basic / Pro tiers)

`modern_grooming_authority/DESIGN.md` is the design system backing all screens: "Nocturnal Premium" —
dark charcoal surfaces, gold/brass (`#f2ca50`/`#d4af37`) primary accent, Hanken Grotesk typeface, 0.25rem
button/input radius, 0.5rem card radius, 8px spacing base, 20px container margins. Reuse these tokens
(colors, spacing, radii, type scale) when building the corresponding real pages in `src/app/` so the
implemented UI matches the mockups rather than reinventing a palette.

No routes, components, or Prisma models yet exist for any of this (still just the `Post` scaffold) — search,
booking, barber profiles/dashboard, and subscription plans are all unbuilt features implied by these mockups.
